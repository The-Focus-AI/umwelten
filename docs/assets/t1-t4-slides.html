T3

LLM-First
Credential Management

Re-scoped — no custom HTML UI. Credentials are managed through Gaia chat using the standard container UI's tool-call rendering.

User
Add a credential for our QuickBooks read-only access
Gaia

I'll add that credential to the catalog.

add_credential {
  name: "quickbooks-read-key",
  label: "QuickBooks Read-Only",
  provider: "intuit/quickbooks",
  capabilities: ["quickbooks:read"],
  scopes: ["accounts:read"]
}

Added credential "quickbooks-read-key" (intuit/quickbooks).

Design decision

The LLM is the interface. Tool calls and results render inline in the standard chat view — no bespoke HTML tabs needed. Credential operations are discoverable through natural language.

Tool coverage

All four T2 tools are immediately available in Gaia chat: add, list (with provider & capability filters), remove, and verify credentials — all managed by the LLM agent.

architecture

How the pieces fit together

1

Master Vault

GaiaSecretVault stores actual secret values — API keys, tokens, credentials — in a single 0600-mode JSON file. Names are lookup keys.

GITHUB_TOKEN
QUICKBOOKS_KEY
TAVILY_API_KEY
2

Credential Catalog

CredentialCatalog stores metadata about each secret: provider, capabilities, scopes, status, and verification timestamps — no values.

name: "gh-read-key"
provider: "github"
caps: ["github:read"]
status: "active"
3

Capability Resolver

CapabilityResolver bridges the two: given a config's capability bindings, looks up credentials in the catalog, validates them, and pulls secrets from the vault.

capability → credential
→ catalog check
→ vault read
→ secrets.json

create_habitat → buildSeedFiles → seedVolume → startContainer
