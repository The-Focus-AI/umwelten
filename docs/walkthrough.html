T2: Credential Catalog — Data Model, File Storage, CRUD Tools CORE ▶

Goal

Add a structured credential catalog to Gaia's data directory. The catalog stores metadata about every secret in Gaia's master vault — what each key is, what capabilities it grants, what provider it's for, and when it was last verified. No actual secret values are stored in the catalog.

What was built

  • CredentialEntry type: name, label, provider, capabilities, scopes, dashboardUrl, sourceVaultRef, status, lastVerified, refreshTokenExpiry
  • CredentialCatalog class: load/save from credentials.json, add/remove/get/list/listByCapability/listByProvider/verify
  • Gaia tools: add_credential, list_credentials, remove_credential, verify_credential
  • Duplicate credential names rejected on add
  • 23 unit tests in credential-catalog.test.ts

How to validate

  1. Start Gaia: 
    dotenvx run -- pnpm run cli habitat gaia -p google -m gemini-3-flash-preview
  2. In the Gaia web UI chat (or MCP), add a credential: 
    add_credential name="github-read-key" label="GitHub Read Key" provider="github" capabilities=["github:read"] scopes=["repo:read"]
  3. List credentials: 
    list_credentials
    → Should show the github-read-key with status=unknown
  4. Verify the credential: 
    verify_credential name="github-read-key"
    → Status should change to active, lastVerified timestamp populated
  5. Check the on-disk file (no secret values): 
    cat gaia-data/credentials.json | python3 -m json.tool
  6. Remove the credential: 
    remove_credential name="github-read-key"
  7. Run unit tests: 
    pnpm test:run packages/habitat/src/gaia/credential-catalog.test.ts

T5: Gaia Tools for Capability Bindings TOOL ▶

Goal

Gaia tools that let operators bind and unbind capabilities to habitats. Extend create_habitat to accept an optional capabilities parameter.

What was built

  • create_habitat extended with capabilities parameter — validates against catalog
  • bind_capability tool — validates credential + capability, adds to habitat config, re-seeds volume
  • unbind_capability tool — removes capability binding, re-seeds volume
  • list_habitat_capabilities tool — shows all bindings with credential status
  • Rejects bindings where credential doesn't exist or doesn't grant the capability
  • 6 unit tests in gaia-tools-capability.test.ts

How to validate

  1. Start Gaia and add a credential in the catalog: 
    add_credential name="gh-read" label="GitHub Read" provider="github" capabilities=["github:read"]
  2. Store the actual secret in the master vault: 
    set_secret name="gh-read" value="ghp_xxxx"
  3. Create a habitat with a capability binding: 
    create_habitat id="test-bot" name="Test Bot" provider="google" model="gemini-3-flash-preview" capabilities=[{capability: "github:read", credential: "gh-read"}]
  4. List capability bindings: 
    list_habitat_capabilities habitatId="test-bot"
  5. Try binding an invalid credential — should get a validation error: 
    bind_capability habitatId="test-bot" capability="quickbooks:read" credential="gh-read"
    → Should fail: "Credential 'gh-read' does not grant capability 'quickbooks:read'"
  6. Unbind a capability: 
    unbind_capability habitatId="test-bot" capability="github:read"
  7. Run unit tests: 
    pnpm test:run packages/habitat/src/gaia/gaia-tools-capability.test.ts

T8: Standards Broadcast Tool (A2A Audit Trigger) TOOL ▶

Goal

A Gaia tool that sends an A2A message to one or all running habitats telling them to pull the latest standards and run a self-audit.

What was built

  • broadcast_standards Gaia tool — takes optional habitatId
  • Uses A2A blocking send-message (return_immediately: false)
  • 60-second per-habitat timeout via Promise.race
  • Parallel audits via Promise.allSettled
  • Filters to running habitats with a standards agent
  • Skips non-running habitats and those without standards agent (with warning)
  • Returns markdown-structured summary: responded, unresponsive, skipped counts + per-habitat findings
  • Unit tests in gaia-tools-standards.test.ts

How to validate

  1. Create and start a habitat with a standards agent (via T7's auto-seed)
  2. In Gaia chat, run: 
    broadcast_standards
  3. Should return a markdown report with sections per habitat:
    • Responded count, unresponsive count, skipped count
    • Per-habitat findings: compliant items, non-compliant with severity
  4. Test single-habitat mode: 
    broadcast_standards habitatId="audit-me"
  5. Habitats without a standards agent should show as "skipped"
  6. Non-running habitats should show as "unresponsive"
  7. Run unit tests: 
    pnpm test:run packages/habitat/src/gaia/gaia-tools-standards.test.ts

How to validate

  4. Click to expand a habitat — see the list of capability gaps with severity badges

  6. Select a credential and confirm — capability is bound, gap is dismissed
  7. If the habitat is running, a "needs rebuild" warning appears
  8. Click Deny — enter optional reason, gap is dismissed

T12: fnox Integration for Gaia's Own Secrets INFRA ▶

Goal

Integrate fnox as Gaia's secret resolution layer. Gaia resolves its master vault secrets via fnox at container start, with no plaintext secrets baked into the image.

What was built

  • FnoxResolver class in packages/habitat/src/gaia/fnox.ts
  • Detects fnox CLI, writes fnox.toml template on first Gaia boot
  • Resolves secrets via fnox export --format json
  • Falls back to .env/process.env when fnox not installed
  • Detects bootstrap tokens: OP_SERVICE_ACCOUNT_TOKEN, FNOX_AGE_KEY, FNOX_AGE_KEY_FILE
  • Habitats never call fnox — Gaia is the sole broker
  • fnox is NOT bundled into the habitat Docker image
  • 26 unit tests in fnox.test.ts

How to validate

  1. Run unit tests: 
    pnpm test:run packages/habitat/src/gaia/fnox.test.ts
  2. Start Gaia — check console output for fnox mode: 
    dotenvx run -- pnpm run cli habitat gaia -p google -m gemini-3-flash-preview
    → Should log: "[gaia] Secrets resolved via env" (or "via fnox" if fnox is installed)
  3. Check that fnox.toml template was written: 
    ls -la gaia-data/fnox.toml
  4. If fnox is installed: the template should be populated; secrets resolved via fnox
  5. If fnox is NOT installed: template still written, secrets loaded from env
  6. Verify the fnox.toml template has the expected secret declarations: 
    cat gaia-data/fnox.toml
T13: Credential Operation Logging META ▶

Goal

Add timestamped logging for credential-related operations in Gaia. Every add, remove, verify, bind, or unbind of a credential is recorded in a JSONL audit log.

What was built

  • CredentialAuditLogger class — JSONL append-only log at <dataDir>/credential-audit.jsonl
  • Records: add_credential, remove_credential, verify_credential, bind_capability, unbind_capability
  • Each entry: { timestamp, operation, habitatId?, capability?, credential? }
  • read_credential_audit_log Gaia tool — returns most recent N entries (default 50)
  • GET /api/audit endpoint in routes
  • 15 unit tests in credential-audit.test.ts

How to validate

  1. Run unit tests: 
    pnpm test:run packages/habitat/src/gaia/credential-audit.test.ts
  2. Start Gaia and perform some credential operations (add, verify, bind): 
    add_credential name="test-key" label="Test" provider="github" capabilities=["github:read"]
verify_credential name="test-key"
remove_credential name="test-key"
  3. Read the audit log via the tool: 
    read_credential_audit_log
    → Should show: add_credential, verify_credential, remove_credential with timestamps
  4. Check the API:

  5. Check the on-disk file is valid JSONL:
